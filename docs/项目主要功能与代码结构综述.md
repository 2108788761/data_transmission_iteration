## 项目主要功能与代码结构综述

本项目是一套 **压裂施工数据传输客户端**，运行于现场电脑，通过串口 / TCP 从仪表车和供水系统采集施工数据，缓存到本地 SQLite 数据库，并通过 HTTP / WebSocket 上传到远程服务器，供调度中心和数据中心实时查看与存档。

本文从「新手维护」视角，按功能和代码模块进行梳理，帮助快速上手阅读与修改代码。

---

## 一、整体功能概览

- **用户登录与鉴权**
  - 通过账号 / 密码 / 油田信息请求后端 `/api/v2/token/`，获取 JWT。
  - 后续所有 API 和 WebSocket 连接都依赖该 JWT 进行鉴权。

- **井信息配置与状态管理**
  - 在 GUI 中配置区块、井台、井号、压裂次数、层位、段/级、施工队、仪表车厂家、传输方式、设备号。
  - 点击「上传井信息」后，调用后端接口写入 / 查询井段状态码（如 1002 可施工、1006 已结束），并在前端控制后续 Tab 是否可用。

- **两种数据来源模式**
  - **软件直传**：直接读取现场已有的数据库或表格文件（如 `.mdb` / `.db` / `.xlsx`），把表头映射为系统标准字段，再上传。
  - **串口转传**：从仪表车 / 网关串口实时接收秒点数据、粘度/密度数据，先写入本地 SQLite，再按指针上传。

- **多类数据采集与上传**
  - **施工秒点数据**：压力、排量、砂比等主施工数据。
  - **液体性能数据（粘度 / 密度）**：通过 Modbus RTU 从液体性能设备读取寄存器，并按液体类型校准。
  - **供水数据**：通过 Modbus TCP 从供水设备读取瞬时流量与累积液量。

- **本地缓存与断点续传**
  - 现场数据统一写入 `C:/a_transmission_data/.../*.db`（每井段一个库）。
  - 上传指针 `point_1 / point_2`（以及粘度、供水的指针）持久化到 `config.ini`，实现断线 / 重启后的断点续传。

## 二、代码模块与职责划分

### 2.1 入口与登录相关

- **`Run.py`**
  - 程序入口，仅做一件事：创建 `QApplication`，实例化并展示 `LoginWindow`，然后进入事件循环。

- **`LoginWindow.py`**
  - 界面来源：`login.ui` → `login.py` 中的 `Ui_MainWindow`。
  - 主要逻辑：
    - 初始化登录界面：账号、密码、油田下拉框（默认填充如「中石油煤层气公司」）。
    - 点击登录按钮：
      - 可选：获取本机 CPU 序列号和 MAC（目前只打印日志）。
      - 校验必填项。
      - 请求 `POST http://39.101.202.11/api/v2/token/`（或本地/内网地址），传入 `username、password、company`。
      - 判断响应：
        - 若返回 JSON 中含业务 `code` 字段 ⇒ 视为认证失败，弹出警告。
        - 否则从 `access` 字段提取 JWT，赋给 `self.jwt_token`。
      - 登录成功后，创建并显示 `OpenExeWindow(self.jwt_token)` 主窗口，关闭登录窗口。

### 2.2 主业务窗口与 UI

- **`uploadMainWindow.ui` / `uploadMainWindow.py`**
  - 使用 Qt Designer 绘制的主窗口界面，对应的 Python 代码自动生成。
  - 只包含界面元素（各种 `QLabel` / `QComboBox` / `QTabWidget` / `QTextEdit` 等），不包含业务逻辑。

- **`OpenExeWindow.py`**
  - 核心业务窗口类：`OpenExeWindow(QMainWindow)`。
  - 构造函数中做的关键事情：
    - 通过 `uploadMainWindow.Ui_MainWindow` 构建 GUI。
    - 保存从登录窗口传入的 `jwt_token`，后续调用所有后台接口。
    - 初始化一些成员变量：当前 db 文件路径、上传指针、字段配置、井号缓存、`LocalPlotWidget` 等。
    - 调用 `init_data()` 向后端拉取基础信息（区块、井台、设备、仪表车厂家、字段中文标签等），填充下拉框。
    - 调用 `load_config()` 从 `config.ini` 恢复上次的井信息与指针。
    - 动态按业务需求构建 Tab2 / Tab3 / Tab5 内的布局与控件（字段映射、串口选择、按钮等）。
    - 初始化「本地秒点曲线」Tab，并集成 `LocalPlotWidget` 画本地 LogData 曲线。
  - 重要方法概览：
    - **配置读写**
      - `load_config()`：从 `config.ini` 读指针（秒点 / 粘密 / 水）、基本井信息、仪表车字段映射。
      - `save_config()`：将当前指针、井信息、仪表车字段映射回写 `config.ini`，用于断点续传。
    - **与后端交互**
      - `base_API(dic_)`：统一封装 HTTP 请求，自动加上 `Authorization: Bearer <JWT>`。
      - `/basic_info_get`、`/get_well`、`/get_layers`、`/get_period` 等接口用于填充井信息联动级联。
      - `/crew_wellinfo_state_get`、`/temp_db_build`、`/temp_db_build_two`、`/temp_db_build_water`、`/update_state_code` 等用于井段状态与后端临时库维护。
    - **井信息与 Tab 权限**
      - `upload_wellinfo()`：上传当前井段信息，解析状态码，只有拿到 `1002` 才允许进入 Tab2 / Tab3 / Tab5。
      - `tab_clicked()`：在切换 Tab 时检查 `self.state_code`，若未 `1002` 则强制回到井信息 Tab。
    - **软件直传（Tab2）**
      - `openFileDialog()`：选择现场仪表车数据库文件（.mdb / .db / .xlsx 等），更新 `self.file_path`。
      - `get_columns()`：根据仪表车厂家读取现场库表头，填充字段映射下拉框；至少需要映射「套管压力」「套管排量」「砂比」。
      - `start_upload()`：构造 columns 信息调用 `temp_db_build` 在后端创建临时库，然后启动秒点上传线程 `WorkerThread`。
      - `end_upload()`：停止上传线程、重置指针与配置。
    - **串口转传秒点（Tab3 上半部分）**
      - `get_all_ports()`：获取当前系统所有串口。
      - `start_receive()`：根据用户选择的字段与仪表车厂家，创建 / 打开 `C:/a_transmission_data/{井号}/{井段}.db` 的 `LogData` 表，然后启动 `ReceiverThread` 从串口写入秒点数据。
      - `stop_receive()`：停止 `ReceiverThread`。
    - **串口转传粘度/密度（Tab3 下半部分）**
      - `receive_viscosity_data()` / `stop_receive_viscosity_data()`：为选定井段启动 / 停止 `ReceiverViscosityThread`，将粘度/密度写入同一 db。
      - `start_upload_viscosity_data()` / `end_upload_viscosity_data()`：建粘密临时库并启动 `UploadViscosityThread` 进行上传。
    - **供水数据（Tab5）**
      - `receive_water_data()` / `stop_receive_water_data()`：创建 `WaterData` 表并启动 `ReceiverWaterThread`，从指定 IP:端口读 Modbus TCP，并写入 `WaterData`。
      - `start_upload_water_data()` / `end_upload_water_data()`：同样先调用 `temp_db_build_water` 再开启 `UploadWaterThread` 进行上传。
    - **上传线程管理**
      - `start_thread()` / `start_viscosity_data_thread()` / `start_water_data_thread()`：根据当前井段信息和指针创建并启动上传线程。
      - `thread_finished*()` / `update_send_mes*()` / `update_status*()`：响应各线程发来的信号（出错、进度、连接状态），更新指针、状态栏并做自动重连。
    - **本地秒点曲线**
      - `_init_local_plot_tab()`：创建「本地秒点曲线」Tab，并放入 `LocalPlotWidget`。
      - `update_local_plot_source()`：当井段或 db 文件改变时，更新 `LocalPlotWidget` 使用的 db 路径和井段信息。
      - `on_local_plot_refresh_clicked()`：刷新按钮触发，从当前 LogData 读取全部数据并画出多条曲线。

### 2.3 线程实现（`thread.py`）

`thread.py` 集中实现所有长耗时及 I/O 密集逻辑，避免阻塞 GUI 主线程。

- **秒点上传线程：`WorkerThread`**
  - 通过 `QWebSocket` 连接 `ws://39.101.202.11/ws/SecondPointDataTransfer/`，在 `Sec-WebSocket-Protocol` 头中携带 JWT。
  - 使用 `point_1` / `point_2` 作为上传进度指针：
    - 首次运行时，若 `point_1 == 0`，从 `LogData` 中找出最小 / 最大 ID 作为初始范围。
    - 每次从本地 `LogData` 中查询 `ID BETWEEN point_1 AND point_2` 的记录。
    - 特殊处理含 `viscosity` / `density` 的行：根据 `system_time` 合并到对应秒点行，再过滤空 Time 行。
    - 将井信息添加到每条记录前，序列化为 JSON，通过 WebSocket 发送。
    - 服务器返回 `flag == 1002` 表示接收成功，前移指针并通过信号通知主窗口刷新状态与保存指针；`flag == 1004` 则回退或重试。
  - 断线时使用 `QTimer` 周期性重连，并通过 `connect_status_signal` 将状态反馈给 GUI。

- **秒点接收线程：`ReceiverThread`**
  - 基于 `pyserial` 从选定串口读取数据，根据不同仪表车厂家解析：
    - 杰瑞：逗号分隔，格式为「时间 + N 个字段」，直接按映射字段入库。
    - 三一重工：发送固定 20 个数据，客户端根据配置的字段索引（1–18）选择需要的字段。
    - 宏华、四机厂：逗号或空格分隔，必要时拼接日期与时间。
  - 写入目标表：`LogData(Time, 字段..., viscosity, density, system_time)`。
  - 同时将原始数据流写入 `_AllData.txt` 供排查。

- **粘度/密度接收线程：`ReceiverViscosityThread`**
  - 基于 `serial` + `crcmod` 实现 Modbus RTU 协议，周期性向设备发送读寄存器请求。
  - 根据液体类型（滑溜水、胍胶、交联胶等）从 `config.ini` 中读取校准参数 `k`、`b`，计算真实粘度：`true_viscosity = k * viscosity_raw + b`。
  - 将 `viscosity`、`density`、`system_time` 写入同一 db（或专用表），供 `WorkerThread` 上传时合并。

- **粘度/密度上传线程：`UploadViscosityThread`**
  - WebSocket 路径：`ws://39.101.202.11/ws/ViscosityDataTransfer/`。
  - 逻辑与 `WorkerThread` 类似，只是数据来源表为 `ViscosityData`，并且每条记录前插入井信息。

- **供水接收线程：`ReceiverWaterThread`**
  - 直接使用 `socket` 建立 TCP 连接，构造 Modbus TCP 请求帧。
  - 解析响应中的 2 个浮点数（瞬时流量、累积液量），写入 `WaterData(Time, waterflow, accumulatewater)`。

- **供水上传线程：`UploadWaterThread`**
  - WebSocket 路径：`ws://39.101.202.11/ws/WaterDataTransfer/`。
  - 与其他上传线程类似，按 `water_point_1 / water_point_2` 指针增量上传。

### 2.4 本地秒点曲线绘图（`LocalPlotWidget.py`）

- 基于 **PyQt5 Qt Charts** 实现，使用 `QChart` + `QLineSeries` + `QDateTimeAxis` / `QValueAxis`。
- 支持多条曲线：
  - 套管压力（左侧 Y 轴）。
  - 套管排量、砂比（右侧 Y 轴）。
  - 粘度（独立右侧 Y 轴）。
- `ChartViewZoomPan` 封装鼠标缩放 / 拖拽以及 `R` 键还原视野。
- `load_full_data()`：
  - 从当前 db 的 `LogData` 读取所有记录，按时间字段（`Time` 或 `system_time`）解析为时间戳。
  - 将数据点追加到各曲线并自动设置合适的轴范围。
  - 利用 `set_full_ranges()` 记录完整视野，支持一键还原。

## 三、关键数据结构与文件

- **本地 SQLite 库（示例：`C:/a_transmission_data/{井号}/{井段}.db`）**
  - 表 `LogData`：
    - `ID INTEGER PRIMARY KEY AUTOINCREMENT`
    - `Time DATETIME`
    - 一系列施工字段（套管压力、套管排量、砂比等）
    - `viscosity FLOAT`
    - `density FLOAT`
    - `system_time DATETIME NOT NULL`
  - 表 `ViscosityData`（可选，按粘密线程实现）：
    - `Time / Viscosity / density`。
  - 表 `WaterData`（供水 db）：
    - `Time, waterflow, accumulatewater`。

- **配置文件 `config.ini`（关键 section）**
  - `Parameters`：`point_1` / `point_2`。
  - `viscosity_pointer`：`viscosity_point_1` / `viscosity_point_2`。
  - `water_pointer`：`water_point_1` / `water_point_2`。
  - `basic_information_settings`：上次选中的井信息（区块 / 井台 / 井号 / 压裂次数 / 层位 / 段 / 压裂队）。
  - `measuring_truck_field_settings`：不同仪表车字段映射，如 `field_names_HongHua`、`field_names_JieRui` 等。
  - `calibration_parameter`：液体类型对应的 `k` / `b`。
  - `liquid_list`：液体名称列表。

### 4.1 如何定位某个功能对应的代码

- **登录 / 鉴权问题**：看 `LoginWindow.py`，再查看 `base_API` 的请求头和 URL。
- **井信息、状态码错误**：看 `OpenExeWindow.upload_wellinfo()` 和 `base_API('/crew_wellinfo_state_get')` 的调用。
- **直传字段映射错误**：看 `OpenExeWindow.get_columns()`、`start_upload()`，以及 `config.ini` 中的 `measuring_truck_field_settings`。
- **串口解析问题**：看 `thread.ReceiverThread.run()` 中按厂家分支的逻辑。
- **粘度 / 密度有问题**：看 `thread.ReceiverViscosityThread`、`config.ini` 中 `calibration_parameter`，以及前端 Tab3 粘密相关代码。
- **供水曲线不对**：看 `thread.ReceiverWaterThread`、`UploadWaterThread`、`OpenExeWindow` 中的供水相关方法。
- **本地秒点曲线显示问题**：看 `LocalPlotWidget.py` 以及 `OpenExeWindow._init_local_plot_tab()`、`update_local_plot_source()`。

### 4.1.1 粘度/密度通信错误「未收到响应数据」成因分析

现场经常在「施工秒点数据、液体性能数据状态栏」里看到类似日志：

- `通信错误：未收到响应数据`
- 串口 `COMx` 随后被关闭，粘度长期显示 `0.00 mPa?s`，密度为某个固定值。

结合 `thread.ReceiverViscosityThread.run()` 代码，可以还原完整触发路径：

- 该线程在 `run()` 中循环执行：
  - 使用固定串口参数（波特率 9600、8N1、`timeout=1` 秒）打开用户在界面中选择的串口号。
  - 构造一帧 Modbus RTU 请求（设备地址默认为 `0x01`，功能码 `0x03`，起始寄存器 `0x0001`，读取寄存器数量 `0x0005`），并通过 `ser.write(request_frame)` 发送给粘度/密度仪表。
  - 随后调用 `response = ser.read(1024)` 等待回包，如果在超时时间内一个字节都收不到，则 `response` 为空。
  - 一旦 `not response`，代码会抛出 `TimeoutError("未收到响应数据")`，被外层 `except` 捕获后，通过 `update_textEdit` 信号把「通信错误：未收到响应数据」输出到状态栏。
  - 在连续异常后，`finally` 分支会执行 `ser.close()`，于是状态栏又会看到「串口 COMx 已关闭」。

因此，这个错误本质上不是 SQLite 或粘度计算逻辑的问题，而是：**串口 Modbus 请求确实发出去了，但在设定的超时时间内设备没有任何响应**。常见根因包括但不限于：

- 串口号选择错误（界面选中的 `COMx` 并不是粘度仪器实际连接的端口，或该端口正被其他软件占用）。
- 串口参数与设备不匹配（波特率、数据位、校验位、停止位与仪表出厂设置不同）。
- 物理层或接线问题（仪表未上电、RS‑485 A/B 接反、无共地、线缆/转换器故障等），导致请求帧根本没到设备或回包进不了 PC。
- Modbus 协议参数不一致（设备地址不是 `1`，或粘度/密度寄存器不在 `0x0001` 起始、`0x0005` 长度范围内），设备收到帧后直接丢弃。
- 超时时间过短（`timeout=1` 秒在部分链路下可能来不及收到完整响应），造成误判为「未收到数据」。

排查和修复时，建议按下面顺序进行：

- **确认串口号与占用情况**：在设备管理器中找到仪表对应的 COM 口，确保与界面选择一致，且没有其他上位机或串口助手正在占用该口。
- **核对串口参数**：对照粘度/密度仪表说明书，检查波特率、数据位、校验位、停止位设置是否一致，如不符需在 `ReceiverViscosityThread.run()` 中修改。
- **使用独立工具验证设备**：用通用串口调试工具或 Modbus 调试器，在相同串口/波特率下发送相同地址和寄存器的请求，确认仪表是否能正常回包；若在外部工具中也读不到数据，说明问题在设备或接线。
- **校正 Modbus 地址/寄存器**：若说明书中设备地址不是 1，或粘度/密度的寄存器地址与代码默认值不同，需要同步修改 `device_id`、`start_addr`、`reg_count`。
- **视需要调整超时时间**：在确认链路正常后，如仍偶发超时，可适当增大 `timeout`（如 2–3 秒）以提高容错，但不要用过大的值以免卡 UI。

只有当上述硬件与协议参数都正确匹配时，`response = ser.read(1024)` 才能读到非空回包，计算出的 `true_viscosity` 与 `density` 才会持续更新，状态栏中「通信错误：未收到响应数据」提示也会自然消失。

### 4.2 常见扩展点

- **增加新的仪表车厂家协议**
  - 在 `OpenExeWindow.init_data()` / `load_config()` 中增加厂家名称与字段配置。
  - 在 `OpenExeWindow.get_field()` 与串口配置逻辑里加入新厂家对应的 `field_names_XXX`。
  - 在 `thread.ReceiverThread.run()` 中新增厂家分支，按协议解析串口数据并调用 `save_data` 或 `save_data_31`。

- **更换 / 增加后端地址**
  - 所有 HTTP / WebSocket 地址集中在：
    - `LoginWindow.login_success` 中的 `login_url`。
    - `OpenExeWindow.base_API()` 的根 URL。
    - `thread.py` 中各个 WebSocket 的 `API_url` 常量。

- **修改本地 db 路径**
  - 所有 db 和数据文件路径集中在：
    - `OpenExeWindow.start_receive()` / `receive_viscosity_data()` / `receive_water_data()`。
    - `thread.ReceiverThread.save_data*()` / `ReceiverWaterThread` 等保存函数。
  - 如需从 `C:/a_transmission_data/` 改到其他目录，注意统一替换并在测试环境验证。

### 4.3 调试建议

- 启动时先用开发环境直接运行 `Run.py`，配合 `print` / `QMessageBox` 查看关键变量。
- 串口 / TCP 通信问题建议优先用串口助手、Modbus 调试工具确认设备侧数据是否正常，再回到程序中抓解析逻辑。
- 出现上传卡住时，可查看：
  - `config.ini` 中的指针是否仍然在旧值。
  - `E:/data.txt` 或 `_AllData.txt` 中是否有写入。
  - 后端 WebSocket 是否主动断开或返回错误状态码。

## 六、附录：`ReceiverViscosityThread` 中 `time.sleep(0.01)` 与错误定位

### 6.1 `time.sleep(0.01)` 是做什么的？

在 `thread.py` 的 `ReceiverViscosityThread.run()` 中，核心循环如下（只保留关键部分）：

```python
while self.running:
    time.sleep(0.01)
    try:
        data_list = []
        ser.write(request_frame)          # 发送 Modbus RTU 请求帧
        response = ser.read(1024)        # 等待响应（受 timeout=1 控制）
        if not response:
            raise TimeoutError("未收到响应数据")
        ...
    except Exception as e:
        self.update_textEdit.emit(f"通信错误: {str(e)}")
```

- 这里的 `time.sleep(0.01)` 表示：每次循环先休眠 **10 毫秒**，再发送下一帧请求。
- 作用：
  - 降低轮询频率，避免在 CPU 很快的情况下“紧紧地跑死循环”疯狂发包。
  - 给系统和设备一个极短的缓冲间隔，防止连续请求过于密集。
- 实际上，本线程的主要时间开销在 `ser.read(1024)` 上，因为串口对象配置了 `timeout = 1` 秒——如果设备没回包，`read` 最长会等 1 秒才返回空。这意味着整体循环频率主要由超时时间决定，**10ms 的 sleep 几乎不会影响是否报错**。

### 6.2 与「通信错误：未收到响应数据」有关系吗？

结合代码逻辑与现场截图：

- 错误提示来源：
  - 当 `response = ser.read(1024)` 返回空（即在 1 秒超时内没有收到任何字节）时，代码执行  
    `raise TimeoutError("未收到响应数据")`。
  - 该异常被外层 `except Exception as e:` 捕获，并通过  
    `self.update_textEdit.emit(f"通信错误: {str(e)}")` 输出到状态栏，因此你在界面上看到「通信错误：未收到响应数据」。
- 触发这一链路的直接原因是：**在设定的超时内串口没有任何响应数据**（设备没回包 / 接线或参数不匹配），而不是因为 `time.sleep(0.01)`。
- 换句话说：
  - 即便把 `time.sleep(0.01)` 改成 `time.sleep(0.1)` 或删掉，只要设备仍然不回包，`ser.read(1024)` 依旧会超时返回空，错误照样会出现。
  - `sleep` 只影响「多长时间发一次请求」，不会导致「这一次请求是否收到响应」。

### 6.3 具体是哪段代码引起错误提示？

综合上面的分析，截图中的错误信息对应的关键代码链路是：

1. `ReceiverViscosityThread.run()` 中：
   - `response = ser.read(1024)`
   - `if not response: raise TimeoutError("未收到响应数据")`
2. 同一函数的异常处理：
   - `except Exception as e: self.update_textEdit.emit(f"通信错误: {str(e)}")`

因此，**真正引发错误提示的，是这段对 `response` 为空的判断与随后的 `TimeoutError` 异常抛出**，背后的根因仍是串口通信链路（串口号、波特率/校验位、设备地址、寄存器地址、接线、上电等）不匹配或物理层异常，而不是 0.01 秒的 `sleep`。***

## 七、错误归因：代码问题还是现场设备问题？

### 7.1 从代码角度看，这一块逻辑是否有明显 bug？

结合 `ReceiverViscosityThread.run()` 的实现，可以看到当前整体流程是：

1. 按固定参数打开串口：`port_name`、`baudrate=9600`、`8N1`、`timeout=1` 秒。
2. 构造一个标准的 Modbus RTU 请求帧：
   - `device_id = 0x01`
   - `func_code = 0x03`（读保持寄存器）
   - `start_addr = 0x0001`
   - `reg_count = 0x0005`
   - 再加上 CRC。
3. 循环中每轮：
   - `ser.write(request_frame)` 发送请求。
   - `response = ser.read(1024)` 等待响应，如果在 `timeout` 内读不到任何字节就视为超时。
   - 对返回帧做最基本的合法性检查（长度、设备号、功能码），再解析粘度和密度，并保存到本地 db。

从“代码正确性”角度看，这段逻辑是**合理且自洽的**：

- 使用了标准的 `pyserial` 接口，异常也都有捕获。
- 产生错误提示的地方仅仅是在“没有任何返回数据时主动抛超时异常”，这在串口通信里是常见且必要的防御策略。
- 不存在明显的数组越界、死循环、解包错误等“代码 bug”。

换句话说：**程序在“没有收到任何响应”时报错，本身是正确且预期的行为**。

### 7.2 为何现场会频繁出现「未收到响应数据」？

真正决定“有没有响应”的，是**现场链路和协议匹配情况**，而不是 Python 逻辑本身。结合代码中的硬编码参数，可以把风险点拆开看：

- 串口层面的假设：
  - 程序假设所有粘度仪表的串口参数都是：`9600-8N1`、超时 1 秒。
  - 如果某台设备实际出厂配置不是这个组合（例如 4800 波特率、带奇偶校验），即使线路完全正常，设备也会**无法正确识别请求帧并且不会回包**。

- Modbus 协议层面的假设：
  - 程序假设设备地址是 `1`，粘度/密度寄存器位于 `0x0001` 开始长度 `0x0005` 的区间内。
  - 如果现场设备地址被改成了 2/3，或者寄存器映射不同，设备会把当前帧当成“发给别人的”或“非法地址”，同样不会回包。

- 物理层 / 运行环境问题：
  - 串口号选错（界面选了 COM7，设备实际挂在 COM5）。
  - 串口被其他程序占用（原厂上位机、串口调试工具一直打开）。
  - 设备未上电、A/B 线反接、没有共地、转接头故障等。

在这些情况下，即便代码逻辑再完美，**`ser.read()` 看到的都是“什么都没有”**，于是走进 `if not response: raise TimeoutError(...)` 分支，状态栏就会不停出现你截图中的那一串“通信错误：未收到响应数据”。

综合来看，目前场景下问题更大概率是：

- **现场设备 / 参数配置 / 接线问题，或者设备协议与代码中的硬编码假设不一致**，而非“Python 代码写错”。

### 7.3 需要改代码吗？什么时候值得改？

从“让程序更健壮、更易适配不同设备”的角度看，有两类改动是**可选的增强**，而不是“必须立刻修复的 bug”：

- **把硬编码的 Modbus / 串口参数抽到配置文件**
  - 例如在 `config.ini` 新增 section（如 `viscosity_serial` / `viscosity_modbus`），配置：
    - 串口波特率 / 校验位 / 停止位。
    - 设备地址 `device_id`。
    - 起始寄存器 `start_addr`、寄存器数量 `reg_count`。
  - 这样不同现场只需要改配置，不用改代码并重新打包，更利于维护。

- **优化错误提示与重试策略**
  - 目前每次超时都会打印“通信错误：未收到响应数据”，在设备长期离线时会刷屏。
  - 可以考虑：
    - 只在第一次超时时打印完整错误，后续间隔性提示。
    - 增加简单的“重连/重新打开串口”逻辑，只在确认连续失败一定次数后才关闭串口。

但即便做了这些增强，**如果现场的串口参数 / 设备地址 / 寄存器配置 / 物理接线依旧不正确，程序仍然会合理地报“未收到响应数据”**——只是报错会更易读、可配置性更好而已。

因此，对当前你遇到的这类错误，**优先建议从现场参数与接线排查入手**，确认在独立 Modbus 调试工具中能稳定读到粘度/密度数据之后，再根据需要考虑是否按上面思路把参数做成可配置，而不是先去大改代码逻辑。***

## 八、截图日志组合问题：多线程混合输出导致的“看起来矛盾”的现象

### 8.1 截图中的关键信息拆解

在第二张截图的「施工秒点数据、液体性能数据状态栏」区域，可以看到类似这样的日志组合（顺序略有简化）：

- `通信错误：未收到响应数据`
- `已停止接收秒点数据！`
- `粘度：0.00 mPa?s, 密度：6.5167 g/cm?`
- `串口 COM7 已关闭。`
- `通信错误：未收到响应数据`
- `数据库文件创建成功，文件名为：...`
- `正在等待串口发送数据`
- `粘度：0.00 mPa?s, 密度：6.5167 g/cm?`
- `通信错误：未收到响应数据`

这些日志实际来自**不同线程**，但都往同一个 `textEdit2` 里写，因此叠在一起看会显得有些“矛盾”。对应关系大致是：

- `已停止接收秒点数据！`：来自秒点接收线程 `ReceiverThread` 的关闭逻辑（`OpenExeWindow.stop_receive()` 中写入）。
- `数据库文件创建成功，文件名为：...`、`正在等待串口发送数据`：来自 `OpenExeWindow.start_receive()` 在创建本地 `LogData` 数据库成功后的提示。
- `粘度：0.00 mPa?s, 密度：6.5167 g/cm?`：来自 `ReceiverViscosityThread` 在成功解析一帧粘度/密度数据后，通过 `update_textEdit` 信号输出的状态。
- `通信错误：未收到响应数据`、`串口 COM7 已关闭。`：来自同一个 `ReceiverViscosityThread.run()` 的异常处理和 `finally` 分支：
  - 当 `ser.read(1024)` 在超时时间内没有读到任何字节时，抛出 `TimeoutError("未收到响应数据")`，被 `except` 捕获并输出「通信错误：未收到响应数据」。
  - 在线程退出或出现串口异常时，在 `finally` 中执行 `ser.close()` 并输出「串口 COMx 已关闭。」。

因此，截图中看起来像是“刚说已停止接收秒点数据，又说粘度在更新，又说 COM7 已关闭，又说在等待串口发数据”的情况，本质上是：

- **秒点接收线程** 和 **粘度接收线程** 分别在不同时间点写入日志；
- 文本区域是共享的，所以这些消息被时间顺序混在一起展示。

### 8.2 这背后到底是什么问题？

从代码逻辑和日志顺序综合判断，实际包含两个层面的现象：

- **现象 A：秒点部分**
  - `数据库文件创建成功 ...` + `正在等待串口发送数据`：说明秒点接收逻辑（`start_receive()` + `ReceiverThread`）这边已经正常完成本地 db 创建并在等现场秒点串口给数据。
  - 如果之后长时间没有“xxx 秒的数据已保存！”之类的日志，只能说明：**现场秒点串口目前还没有真正发数据到程序**（或端口/参数/接线有误），这与前面分析的粘度问题类似，根因在设备 / 线路侧。

- **现象 B：粘度/密度部分**
  - 多次出现「通信错误：未收到响应数据」和「串口 COM7 已关闭。」：这正是 `ReceiverViscosityThread` 在没有读到任何 Modbus 回包、或串口出现异常时的预期行为。
  - 中间偶尔出现一两次 `粘度：0.00 mPa?s, 密度：6.5167 g/cm?`，一般意味着：某次读取有回包，但解析出的粘度值为 0，之后再次通信又超时。

归根结底，这仍然是**现场设备/链路的问题**：

- 秒点串口没有稳定发数据到程序；
- 粘度仪表对应的串口请求发出去了，但设备经常不回包，或参数/地址/寄存器不匹配，导致读不到数据。

程序这边的行为（创建 db、等待数据、在超时时抛出“未收到响应数据”、在关闭时提示串口已关闭）都是符合代码逻辑和预期的。

### 8.3 需要改代码吗？

就这张截图本身来说：

- **不需要为了“消灭这几行日志”去改核心串口逻辑**，因为：
  - 报错条件（没收到任何响应）是真实发生的；
  - 核心问题在于设备端/接线/参数，改代码不能让仪表自动开始回包。
- 若后续从“可维护性 / 体验”角度希望优化，可以考虑：
  - 把粘度串口的设备地址、寄存器等都挪到 `config.ini`，减少因硬编码不匹配带来的问题；
  - 把秒点和粘度的日志输出区分开（例如各用一个 `QTextEdit`），避免新同事误以为这些日志来自同一个线程；
  - 对连续超时的情况压缩提示频率，防止状态栏被相同报错刷屏。

但在你当前遇到的问题上，**优先仍然是按前文第七章的清单，排查设备接线、串口号/波特率/校验位、Modbus 地址与寄存器配置是否与代码中的默认值完全一致**；只有当在独立的串口 / Modbus 调试工具中也无法可靠读到数据时，才能确认这是“现场侧”的问题，而不是程序误判。***

## 九、粘度持续刷新但偶尔出现“通信错误：未收到响应数据”的情况

### 9.1 现象描述

第三张截图中可以看到：

- 状态栏连续打印多条粘度/密度读数，例如：
  - `粘度：1.40 mPa?s, 密度：6.5166 g/cm?`
  - `粘度：1.80 mPa?s, 密度：6.5167 g/cm?`
  - `粘度：1.60 mPa?s, 密度：6.5169 g/cm?`
- 在这些正常读数之间，**偶尔插入一条**：`通信错误：未收到响应数据`。

说明两点：

- `ReceiverViscosityThread` 大部分时间能成功完成“请求 → 接收响应 → 解析 → 写库并输出粘度/密度”的流程；
- 偶尔某一轮循环中，`ser.read(1024)` 在 `timeout=1` 秒内读不到任何字节，于是触发了前文分析过的 `TimeoutError("未收到响应数据")`，但线程没有退出，下一轮仍然继续发送请求并成功收到数据。

换句话说：**这是一个“间歇性超时”的问题，而不是“完全收不到数据”。**

### 9.2 产生这种间歇性超时的常见原因

在这类“有时候能正常读，有时候超时”的场景下，根因通常有几种可能：

- **物理链路质量或干扰问题**
  - RS‑485 总线太长、终端匹配/偏置电阻不理想、接地不良，导致偶发性丢包。
  - 现场电磁环境强（大功率电机、变频器等），在某些时刻串口信号被干扰，导致那一帧的数据接收失败。

- **请求节奏与设备内部采样/处理节奏不匹配**
  - 当前线程每次循环都会发送一帧 Modbus 请求，在 `timeout=1` 秒内等待响应；若设备内部处理较慢或有自带的采样周期，偶尔某一帧来不及处理/回包，就会形成一次“空读”。

- **设备偶发忙碌或错误**
  - 部分仪表在内部执行自检、存储、模式切换时，短时间内不响应外部 Modbus 请求。

- **串口缓冲区/驱动层偶发异常**
  - 在 Windows 上，偶尔会出现驱动层丢包、缓冲区未及时刷新等情况，特别是在多线程高频访问同一串口时更明显。

从当前代码看，`ReceiverViscosityThread` 在超时后只是记录一次“通信错误：未收到响应数据”，不会立刻关闭串口或退出循环，因此你会看到“正常值 + 偶发错误提示”交替出现，这是**预期设计的防御机制**。

### 9.3 如何处理与改善

对于这种“粘度数据总体可用、但间歇性报一次超时”的情况，建议的处理思路是：

- **1）首先确认数据质量是否满足业务要求**
  - 如果实际曲线在 Web 端或本地画图上看起来连续且合理，偶尔一两点空值对整体分析影响不大，那么这种程度的偶发超时在串口通讯里是可以接受的。
  - 若对每一个时间点都有强一致性要求（比如要精确对齐服务器秒点数据），则需要进一步优化链路与代码。

- **2）现场侧可采取的改进措施**
  - 尽量缩短信号走线、确保 A/B 线与地线连接正确，并根据设备厂商建议加终端电阻/偏置电阻。
  - 若有条件，将 RS‑485 线与强电线分开走线，减少电磁干扰。
  - 查阅粘度仪表说明书，了解其推荐的 Modbus 轮询周期（有的厂商建议 200ms/500ms/1s 等），不要轮询得过于频繁。

- **3）代码层面可考虑的温和优化（可选）**
  - **适当延长超时时间**：将 `timeout` 从 `1` 秒调大到 `2` 秒（前提是不会影响整体 UI 响应），对网络或设备偶发抖动有一定容错作用。
  - **降低轮询频率**：在 `while self.running:` 循环中把 `time.sleep(0.01)` 增大到 `0.1` 或更高，让设备有更充分的时间处理上一帧请求，同时减少总线占用率。
  - **压缩错误日志**：例如只在连续 N 次超时时打印一条“通信存在异常，请检查线路/设备”，避免状态栏被重复相同的错误刷屏。

需要强调的是：**当前代码在面对这类偶发超时时的行为是合理的**——它既不会悄悄吞掉问题（而是明确提示“未收到响应数据”），也不会因为一次失败就退出线程；要不要在此基础上微调 `timeout` 和 `sleep`，应根据现场对“实时性 vs 稳定性”的权衡来决定。***

## 十、如何确认粘度/密度数据是否已成功采集并保存到本地

### 10.1 粘度数据写入到哪里？

结合 `ReceiverViscosityThread` 和 `OpenExeWindow.receive_viscosity_data()` 的实现，粘度/密度数据的落库路径是：

- 目标文件：和秒点数据 **共用同一个井段 db 文件**，路径形如：  
  `C:/a_transmission_data/{井号}/{井号}第{压裂次数}次压裂{层位}第{段}段.db`
- 在 `ReceiverViscosityThread.save_data()` 中，使用的是：

```python
conn = sqlite3.connect(self.file_path)
cursor.execute(
    'INSERT INTO LogData (viscosity,density,system_time) VALUES (?,?,?)',
    data
)
```

- 也就是说：每成功从粘度仪表解析出一组数据，就会向当前井段的 `LogData` 表插入一行，只写入三列：
  - `viscosity`：校准后的真实粘度（单位 mPa·s）；
  - `density`：密度（g/cm³，对应代码中的 4 位小数）；
  - `system_time`：采样时间（字符串，格式 `HH:MM:SS`）。
- 秒点数据上传线程 `WorkerThread.check_updates()` 在读取 `LogData` 时，会把这些仅含 `viscosity/density` 的行按 `system_time` 与秒点行合并，最终形成一条同时带压力/排量/砂比与粘密的完整记录。

因此：**只要 `ReceiverViscosityThread` 成功运行，粘度/密度一定是先写在当前井段的 `LogData` 表里，再被上传线程读取。**

### 10.2 从状态栏如何初步判断“已经采集到了”？

在不看数据库文件的情况下，可以先通过 GUI 状态栏做快速判断：

- 当粘度线程正常工作时，`ReceiverViscosityThread.run()` 会频繁调用：

```python
self.update_textEdit.emit(
    f"粘度: {true_viscosity:.2f} mPa?s，密度: {density:.4f} g/cm?"
)
```

- 如果你在「施工秒点数据、液体性能数据状态栏」中持续看到这种格式的日志（并且数值在变化），说明：
  - 串口链路通；
  - Modbus 请求有响应；
  - 粘度/密度已经被成功解析并写入到本地 db（因为 `save_data()` 和 `update_textEdit` 是在同一 try 块内顺序执行的，能看到这行日志就意味着 `INSERT` 已经执行）。

### 10.3 如何用工具 100% 确认本地确实有粘度数据？

如果需要完全确认，可以直接查 SQLite 文件：

1. **找到当前井段 db 文件路径**
   - 根据界面上选的井号、压裂次数、层位、段号，拼出文件名：
     - 例如：`吉深8-5平01井第1次压裂8#煤层第7段` 对应  
       `C:/a_transmission_data/吉深8-5平01井/吉深8-5平01井第1次压裂8#煤层第7段.db`
   - 这是 `OpenExeWindow.start_receive()` / `receive_viscosity_data()` 中构造的同一条路径。

2. **用 SQLite 浏览工具打开**
   - 可以使用 DB Browser for SQLite、SQLiteStudio，或命令行 `sqlite3`。
   - 打开上述 `.db` 文件，查看 `LogData` 表结构和数据。

3. **检查 `LogData` 表中的粘度/密度列**
   - 表头中应包含 `viscosity`、`density`、`system_time` 三列。
   - 在数据视图中，如果看到类似：
     - 某些行只有 `viscosity/density/system_time` 有值、其它字段为空；
     - 或者部分秒点行的 `viscosity/density` 已经被写满（说明合并已发生），  
     则表明粘度/密度已成功采集并写入本地。

4. **脚本快速验证（开发环境）**
   - 也可以在 Python 控制台临时执行类似代码：

```python
import sqlite3
conn = sqlite3.connect(r"C:/a_transmission_data/井号/井段名.db")
cur = conn.cursor()
cur.execute("SELECT id, viscosity, density, system_time FROM LogData WHERE viscosity IS NOT NULL LIMIT 10")
print(cur.fetchall())
cur.close()
conn.close()
```

   - 如果能打印出若干行非空结果，就说明粘度数据已经落库。

### 10.4 与上传/画图的关系

- **上传链路**：`WorkerThread` 在上传秒点数据前，会先从 `LogData` 中把仅含 `viscosity/density` 的行按时间维度合并到对应秒点行，再通过 WebSocket 发送；所以只要本地有这些粘度记录，上传线程就能一并带上。
- **本地秒点曲线**：`LocalPlotWidget.load_full_data()` 会同时读取 `套管压力 / 套管排量 / 砂比 / viscosity / density` 等字段，并在第三根 Y 轴上绘制「粘度」曲线；因此，当你刷新本地曲线 Tab 并看到粘度曲线有变化时，也可以认为粘度数据已经落库并被正确读取。***

## 十一、同时接收秒点和粘度数据时，如何确认秒点数据也在接收？

### 11.1 秒点和粘度的状态栏输出分别来自哪里？

在「施工秒点数据、液体性能数据状态栏」这个文本框（`textEdit2`）中，实际上混合了**两个线程**的输出：

- **秒点接收线程 `ReceiverThread`**
  - 在正常保存一条秒点数据后，会通过 `update_textEdit` 信号触发 `OpenExeWindow.receiver_thread_finished()`：

```python
def receiver_thread_finished(self, data):
    self.ui.textEdit2.append(f"{data}秒的数据已保存！")
```

  - 这里的 `data` 一般是秒点时间（如 `12:00:05`），所以当秒点接收正常时，你应该能在状态栏看到类似：
    - `12:00:05秒的数据已保存！`
    - `12:00:06秒的数据已保存！`

- **粘度接收线程 `ReceiverViscosityThread`**
  - 在成功解析一帧粘度/密度后，会通过 `update_textEdit` 信号调用：

```python
self.update_textEdit.emit(
    f"粘度: {true_viscosity:.2f} mPa?s，密度: {density:.4f} g/cm?"
)
```

  - 所以你看到的 `粘度: X.XX mPa?s, 密度: Y.YYYY g/cm?` 就是粘度线程在往同一个 `textEdit2` 写。

因为这两个线程都往同一个文本框追加内容，如果粘度采样频率较高、而秒点每秒只来一条甚至偶尔缺失，视觉上就容易产生“状态栏里几乎全是粘度日志，好像没有秒点日志”的错觉，但其实两者是**并行输出**的。

### 11.2 如何在界面上确认秒点数据确实在接收？

在不看数据库的前提下，可以从以下几类信息判断：

- **查看是否出现“xx秒的数据已保存！”的行**
  - 这是 `ReceiverThread` 成功写入一条秒点数据后**唯一的明确标记**。
  - 如果在你开始接收一段时间后，状态栏中完全看不到类似“`xx秒的数据已保存！`”的日志，说明秒点接收线程可能：
    - 没有真正启动（例如没有点击「开始接收」或前置条件不满足）；
    - 串口无数据 / 参数不匹配，导致虽然线程在跑，但 `save_data()` 分支一直没有被触发。

- **观察“接收到数据：[...]”类型的日志**
  - 在 `ReceiverThread.run()` 中，当成功从串口解析出一行数据（不一定已经写库）时，会通过 `update_textEdit_2` 信号调用：

```python
def update_textEdit_pointdata(self, data):
    self.ui.textEdit2.append(f"{data}")
```

  - 若仪表车连续发秒点数据，你通常会看到 `接收到数据：[ ... ]` 或类似内容的日志（取决于当前厂家分支）。

只要在状态栏里能看见上述任意一种与秒点相关的行，说明秒点接收线程是活动的、串口也有数据在流入。

### 11.3 秒点数据在界面之外保存在哪里？

- 和前面粘度一样，秒点数据最终都写入当前井段的本地 SQLite 库：
  - 路径：`C:/a_transmission_data/{井号}/{井号}第{压裂次数}次压裂{层位}第{段}段.db`
  - 表：`LogData`
- `ReceiverThread.save_data()` / `save_data_copy()` / `save_data_31()` 会在解析一秒数据后，将该秒的全部字段（时间+套压+排量+砂比+其他选定字段+system_time）插入到 `LogData`。
- 所以如果你想进一步确认秒点是否真正写入，可以：
  - 打开对应 `.db` 文件，查看 `LogData` 表中是否有不断增加的新行；
  - 或者使用文档第 10.3 节中的 Python 示例，把查询列改为 `Time, 套管压力, 套管排量, 砂比, system_time` 等字段做快速检查。

### 11.4 如果状态栏只看到粘度行，看不到“xx秒的数据已保存！”怎么办？

这种情况下，通常说明：

- 粘度串口工作基本正常（能持续收到 Modbus 回包），
- 而**秒点串口还没有真正送数据到程序**，可能原因包括：
  - Tab3 上方“串口号(秒点)”没有选对口或仍为空；
  - 字段序号映射未配置/配置不合法，导致 `ReceiverThread` 在解析时无法满足字段数量校验，从而不调用 `save_data()`；
  - 仪表车侧尚未开始向该串口输出秒点数据（或仍在另一串口）；
  - 串口参数与仪表车不匹配，造成秒点数据无法正确解析。

排查顺序建议：

1. 在界面确认：
   - 已选择正确的“串口号(秒点)”；
   - 字段序号配置无冲突（没有弹“序号选择重复”的警告）；
   - 点击了「开始接收」，按钮变灰、「停止接收」可用。
2. 若仍看不到任何秒点相关日志：
   - 用串口调试助手单独连到该口，看是否有秒点原始数据；
   - 若调试工具也没有数据，则问题在仪表车或接线；
   - 若调试工具能看到数据，再回到程序中重点检查厂家选择、字段映射等逻辑。***

## 十二、错误 “通信错误: no such table: LogData” 的含义与处理

### 12.1 这是什么错误？

在最新截图中，「施工秒点数据、液体性能数据状态栏」反复出现：

- `通信错误: no such table: LogData`

这行日志的来源是我们在多个线程中统一的异常处理逻辑，例如在 `ReceiverViscosityThread.run()` 的 `except` 分支中：

```python
except Exception as e:
    self.update_textEdit.emit(f"通信错误: {str(e)}")
```

当线程内部执行与 SQLite 相关的操作（例如 `INSERT INTO LogData (...) VALUES (...)`）时，如果当前数据库文件中**根本不存在名为 `LogData` 的表**，SQLite 会抛出 `sqlite3.OperationalError: no such table: LogData`，然后被 `except` 捕获并原样拼到状态栏，于是你看到的就是：

- `通信错误: no such table: LogData`

本质上，这是一个**数据库结构错误**：程序假定当前 `.db` 文件里一定有 `LogData` 表，但实际打开的文件里并没有这个表。

### 12.2 为什么会出现 “no such table: LogData”？

结合 `OpenExeWindow` 中各个入口的逻辑，可以总结出几种典型场景：

- **场景 A：只打开了一个已有的 db 文件，但这个文件不是由本软件创建的**
  - 比如直接在「上传数据」Tab 中选择了某个历史现场库，或者手动拷贝/改名了一个 .db 文件，这个文件内部可能叫 `LogData_2024`、`history_data_xx` 等，而**没有名为 `LogData` 的表**。
  - 当粘度或秒点接收线程按我们的固定 SQL `INSERT INTO LogData (...)` 去写入时，就会立刻报 `no such table: LogData`。

- **场景 B：启动了“接收粘度数据”，但对应井段的 db 还没通过“开始接收秒点数据”初始化**
  - 目前粘度接收逻辑假设：对应井段的秒点 db 已经存在，并由 `start_receive()` 创建过 `LogData` 表。
  - 如果只在下半部分（粘/密度）配置好串口后就点击「开始接收」，而上半部分（秒点）从未点击「开始接收」初始化 db，那么：
    - `receive_viscosity_data()` 可能找到的是一个空的 `.db` 文件，或旧版本仅含 `ViscosityData` 表的库；
    - 此时代码直接对 `LogData` 做 `INSERT`，自然会得到 `no such table: LogData`。

- **场景 C：db 文件被外部工具或旧版本程序改过结构**
  - 比如用 SQLite 工具手动删掉了 `LogData` 表，或旧版本程序创建的库结构与当前版本不一致，也会触发同样的错误。

### 12.3 如何确认并修复？

排查步骤建议如下：

1. **确认当前在用的是哪个 db 文件**
   - 根据界面上显示的井号、压裂次数、层位、段号，在 `C:/a_transmission_data/{井号}/` 下找到对应的 `.db` 文件。
   - 这是 `OpenExeWindow.start_receive()` 和 `receive_viscosity_data()` 使用的同一条路径。

2. **用 SQLite 工具查看是否存在 `LogData` 表**
   - 用 DB Browser for SQLite / SQLiteStudio 打开该 `.db`。
   - 在“表结构”或“浏览数据”列表中查看：
     - 是否有名为 `LogData` 的表；
     - 如果没有，说明当前库不是标准的秒点库。

3. **按正确顺序重新初始化**
   - 推荐的正确操作顺序是：
     1. 在 Tab1 完成井信息上传，拿到 1002 状态；
     2. 在 Tab3 上半部分配置好字段 → 点击「开始接收」**秒点数据**，由程序自动创建带有 `LogData` 表结构的 db 文件；
     3. 确认状态栏出现“数据库文件创建成功，文件名为：...”以及“正在等待串口发送数据”等日志；
     4. 然后在下半部分配置粘度串口 → 点击「开始接收」**粘/密度数据**。
   - 若之前已生成了结构不正确的 `.db` 文件，可以备份后删除该文件，让程序重新按照当前版本的表结构创建一个新的秒点库。

4. **避免直接对非本程序创建的 db 使用“接收秒点/粘度”功能**
   - 对外部采集的历史 db、第三方系统产生的 db，更安全的做法是：
     - 要么在“上传数据”（直传）路径中使用（程序会先读表头再由后端建临时库）；
     - 要么在导入前通过脚本先转换为带有 `LogData` 表结构的中间库。

### 12.4 是否需要改代码？

从健壮性角度，可以考虑在写入前增加一次表存在性检查，例如在 `ReceiverViscosityThread.save_data()` / `ReceiverThread.save_data()` 中先执行：

```sql
SELECT name FROM sqlite_master WHERE type='table' AND name='LogData';
```

若不存在则直接给出更友好的提示（如“当前 db 不包含 LogData 表，请先通过‘开始接收秒点数据’初始化该井段！”），而不是让 SQLite 抛出 `no such table`；但这属于**用户体验优化**，并不改变根本要求：**必须使用由本程序按当前版本创建的标准秒点 db（含 LogData 表）来接收和保存数据**。***

## 十三、日志 “接收到数据[...] / 发送端的数据个数与选择的字段不匹配！” 的含义

### 13.1 现在算是“已经接收到秒点数据”了吗？

在你给出的截图中，「施工秒点数据、液体性能数据状态栏」出现了这些组合日志：

- `粘度: 0.00 mPa?s, 密度: 6.5166 g/cm?`
- `接收到数据：['1677.0 0.0 0.0 0.0']`
- `发送端的数据个数与选择的字段不匹配！`

其中：

- 粘度行来自 `ReceiverViscosityThread`；
- `接收到数据：[...]` 和 “发送端的数据个数与选择的字段不匹配！” 来自秒点接收线程 `ReceiverThread.run()` 中对应仪表厂家（如“杰瑞”）的分支：

```python
if ser.in_waiting > 0:
    raw_data = ser.read(ser.in_waiting)
    data = raw_data.decode('utf-8')
    buffer += data
    lines = buffer.splitlines(keepends=True)
    for line in lines[:-1]:
        line = line.strip()
        if line:
            data_list = line.split(',')
            self.update_textEdit_2.emit(f'接收到数据：{data_list}')
            ...
            if len(data_list) == field_num:
                self.save_data(self.field_name, tuple(data_list), field_num)
            else:
                self.update_textEdit_2.emit('发送端的数据个数与选择的字段不匹配！')
```

含义是：

- 串口链路是通的，程序**已经收到了秒点原始数据行**（如 `'1677.0 0.0 0.0 0.0'`），并成功解析成 `data_list`，通过 “接收到数据：[...]” 打印出来。
- 但由于 `len(data_list)` 与程序期望的 `field_num` 不相等，这行数据**暂时没有写入本地 `LogData` 表**，而是输出了“发送端的数据个数与选择的字段不匹配！”。

所以，从“是否接收到串口数据”角度：**是的，秒点数据已经进程序了**；但从“是否成功保存到本地数据库”角度：**目前这几秒的数据还没有落库**。

### 13.2 “数据个数与选择的字段不匹配”具体指什么？

在 `ReceiverThread.run()` 的前面有：

```python
# 发送的字段数量 还需要加上时间字段，字段数量+1
field_num = len(self.field_name) + 1
...
filtered_list = [{'label': item['label'], 'value': int(item['value'])} ...]
field_name = [item['label'] for item in filtered_list]
```

- `self.field_name`：你在 Tab3「字段设置」里勾选的、希望保存到本地库中的字段（套管压力、套管排量、砂比等）。
- `field_num = len(field_name) + 1`：程序对**每行串口数据字段个数的期望值**：1 个时间字段 + 你勾选的这些字段数。

当真正读到一行数据时：

- 程序把这一行按逗号或空格拆成 `data_list`；
- 如果 `len(data_list) == field_num`：
  - 认为“发送端的字段个数”和“本地想保存的字段个数（+时间）”一致，调用 `save_data()` 写入 `LogData`；
- 如果不等：
  - 认为“发送端当前发的列数”和“界面上配置的目标字段数”不一致，为避免错位写库，直接提示“发送端的数据个数与选择的字段不匹配！”，并且**不保存这行数据**。

常见导致不匹配的情况：

- 仪表车只发了 `时间 + 3 个数`，但你在 Tab3 里勾选了 4~5 个字段；
- 仪表车发的数据格式里有额外的逗号 / 空白，拆出来的 `data_list` 比预期更长；
- 对不同厂家，协议格式不同（逗号/空格、是否包含日期），而界面上的配置与实际协议不一致。

### 13.3 如何调整才能既“接受到”又“成功保存”秒点数据？

目标是让 `len(data_list) == field_num` 成立。建议步骤：

1. **用串口调试工具确认实际数据格式**
   - 直连秒点串口，看一行原始数据长什么样：
     - 一行有几个数？
     - 分隔符是逗号还是空格？
     - 时间是否单独一列？

2. **回到 Tab3 调整字段勾选**
   - 在“字段设置”区域，只勾选与你实际看到的字段数匹配的那几个字段；
   - 注意：`实际每行总字段数 = 1（时间） + 你勾选的字段数`。

3. **确保厂家选择与协议一致**
   - 如果仪表车厂家改了（杰瑞 / 三一 / 宏华 / 四机厂），要确保：
     - 界面上“仪表车厂家”选择正确；
     - 若是三一重工，还要确保字段序号（1~18）设置正确且无重复。

4. **重新观察状态栏**
   - 调整正确后，日志应从：
     - `接收到数据：[...]; 发送端的数据个数与选择的字段不匹配！`
   - 变为：
     - `接收到数据：[... ]`
     - `12:00:05秒的数据已保存！`

这说明：秒点串口数据不仅被接收到了，而且已经按当前字段映射成功保存到 `LogData` 表，可供后续上传和本地秒点曲线使用。***
